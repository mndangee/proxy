# 프로젝트 전체 동작 사용 가이드 (프록시 + 업스트림)

이 가이드는 **화면(업스트림 프로젝트) + API 모의응답(DataForge 프록시)**를 함께 사용하는 흐름 기준입니다.

## 언제 이 모드를 쓰나

- 실제 프로젝트 화면을 띄운 상태에서 API 응답만 바꿔 테스트하고 싶을 때
- 일부 API는 모의 응답으로, 나머지는 업스트림 서버로 그대로 흘려 보내고 싶을 때

## 개념 한 번에 정리

- **프록시(모의 서버)**: 저장된 mock 응답을 먼저 반환
- **업스트림 서버(프로젝트 서버)**: mock이 없는 요청을 처리
- **게이트웨이(선택)**: 클라이언트가 고정 포트만 호출할 때 중간에서 라우팅

기본 원칙:
- 클라이언트 URL을 바꿀 수 있으면 게이트웨이 없이도 운영 가능
- 클라이언트 URL이 고정이면 게이트웨이를 켠다

## 권장 포트 예시

- 프록시(모의 서버): `4780`
- 업스트림 서버: `7778`
- 게이트웨이(필요할 때만): `7777`

서로 다른 프로세스이므로 **포트는 겹치면 안 됩니다.**

## 설정 순서 (전체 동작 기준)

1. 홈에서 `프록시 설정` 열기
2. `모의 서버 사용` ON, 프록시 포트 입력 (예: `4780`)
3. `요청/응답 방식` 선택
   - 레거시 트랜 봉투(`header.tranId`) 기반이면 `legacy-tran-envelope`
   - 일반 REST JSON이면 `generic-json`
4. `업스트림 자동 실행` ON (원하면 수동 실행도 가능)
   - 백엔드 작업 폴더 입력
   - 업스트림 포트 입력 (예: `7778`)
   - 필요 시 `Node 실행 파일 경로` 지정
5. 클라이언트 호출 방식에 따라 분기
   - **A. 클라이언트 API 베이스 변경 가능**: 프록시 주소를 직접 바라보게 설정
   - **B. API 베이스 고정(예: 7777 고정)**: 게이트웨이 ON, 클라이언트는 기존 주소 유지

## A안: 클라이언트 베이스 URL 변경 가능할 때 (더 단순)

- 게이트웨이 OFF
- 클라이언트 API 베이스를 프록시로 지정
  - 예: `http://localhost:4780/api`
- 동작:
  - mock 있음 -> 프록시 응답
  - mock 없음 -> 업스트림으로 전달(설정에 따라)

## B안: 클라이언트 베이스 URL을 못 바꿀 때 (게이트웨이 사용)

- 게이트웨이 ON
- 게이트웨이 클라이언트 포트: 고정 포트(예: `7777`)
- 업스트림 포트: 다른 포트(예: `7778`)
- 프록시 포트: 별도 포트(예: `4780`)
- 동작:
  - 클라이언트는 계속 `http://localhost:7777` 호출
  - 게이트웨이가 요청을 받아 mock 조회 후, 없으면 업스트림으로 전달

## Node 설치 관련 (중요)

- DataForge 앱 자체는 Electron 런타임으로 실행되므로 사용자 PC에 Node가 없어도 실행됩니다.
- 하지만 **업스트림 자동 실행**은 외부 백엔드를 띄우는 기능이라 Node가 필요할 수 있습니다.
- Windows에서는 아래 순서로 Node 실행 파일을 탐색합니다.
  1) `%NVM_HOME%\\v16.x.x\\node.exe`
  2) `C:\\Program Files\\nodejs\\node.exe` (x86 포함)
  3) PATH의 `node`

즉, 업스트림 자동 실행이 필요하면 Node 실행 파일 경로를 지정하거나 시스템에 Node가 있어야 합니다.

## 빠른 동작 점검

1. 프록시 상태가 `running`인지 확인
2. 업스트림 자동 실행이 켜졌다면 업스트림도 `running`인지 확인
3. 테스트 API에 mock 저장 후 호출
4. mock을 지우거나 다른 키로 호출해 업스트림 fallback 동작 확인
5. 실패 시 포트 충돌(`EADDRINUSE`) 먼저 확인

## 자주 막히는 포인트

- 같은 포트를 프록시/업스트림/게이트웨이에 중복 입력
- 레거시 프로젝트인데 `generic-json` 프로필로 실행
- Windows에서 shell 전용 명령(`sh -lc`) 가정한 커맨드 입력
- 업스트림 Node 버전 불일치 (프로젝트 요구 버전과 다름)

필요하면 팀 환경 기준으로 이 문서를 `프로젝트별 템플릿` 형태(포트/명령/프로필 채워진 버전)로 만들어 사용할 수 있습니다.
